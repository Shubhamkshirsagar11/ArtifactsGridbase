Picker spec — how consumers (Rate Builder, bids) select rates. The full hierarchy is: Local Union → Agreement → Rate book → Wage column → Effective date, then classifications from within that book. The breadcrumb always displays whichever levels were real, so every calculation is auditable at a glance. The benefits-package pick stops at the Agreement level — funds are shared across books and columns, so the pay-scale/benefits mix-and-match works unchanged and never asks a book or column question.

Proposed — the Basis picker (decided 7/10)

Read this first — the mental model. There is one way to define a fund's basis: three questions (what the amount multiplies · which hours count · what rate OT/DT hours get). The dropdown's two named options are presets — saved answers to those three questions — because they cover almost every fund. "Custom…" doesn't open a different system: it opens the same three questions, pre-filled with the preset you're on, so you change only what deviates. Flow: pick a preset → done (one click, ~90% of funds). Pick Custom… → a slide-over panel opens "Starting from: [your preset]" → adjust → Done → the cell shows a compact summary with a Custom chip.

After publish

Editing & history

every edit is plain; every edit is logged; nothing is ever locked by the calendar

Once an agreement is published, all changes happen on its detail page (Wages / Benefits / Mapping / Documents / Details tabs — the retired Settings mapping page had been the only exception). Edits apply immediately; there is no locking, no approval chain, and no draft mode for edits. The audit trail is a side effect the user never performs — the system records it.

The seven kinds of edit

1 · Fix a valueclick the cell, type, save — no ceremony. New reads get the corrected number immediately.
2 · Add raise dateappends a period, or splits one mid-schedule. Answers for dates before the new raise date never change. If the date is past the current expiration, the same form requires the new expiration date (typed from the paperwork — never inferred): no agreement may contain a period outside its own term. Rate entry in this flow groups by rate book and carries each wage column, like Step 3.
3 · Classes & fundsadd / rename work as plain edits; remove is an explicit confirmed action, never a silent side effect of saving (today a save that omits a class quietly deletes it — that behavior ends). The same rules cover rate books and wage columns: add/rename plain, remove confirmed.
4 · Mappingssame three-state model as the wizard, edited on the detail page.
5 · Documentsreplace or add source PDFs as they arrive.
6 · IdentityDecided 7/10: name and expiration are directly editable (Details tab, logged). Adding a covered local is allowed — it fans out rate rows for that local. Removing a local is its own confirmed action (it ends coverage, not a field tweak).
7 · Lifecyclesupersede via the checkbox when publishing the successor; "Mark agreement ended…" (admin-only, reason required) for true terminations. The calendar alone never stops rates.
Locals ↔ agreements: many-to-many, both directions
A local can hold several simultaneously active agreements (LU 70's 4-70-A CBA and WFS 4-70-A overlap in the corpus; LU 104 runs North and South side by side), and one agreement can cover many locals (TN's seven, CA's 47 + 1245). Consequences: supersede targets one specific agreement, never a local; status and alerts are per-agreement; consumers always pick an agreement, not just a local. The Review step's "already-covered" notice stays an advisory for exactly this reason — overlap is often legitimate.

The audit trail — four properties

Universalevery write from every surface produces a history entry — which requires closing the side doors (scripts and raw admin edits that leave no trace). An audit trail with exceptions isn't one.
Readablefield-level, human sentences: "Ben changed Journeyman Lineman ST (Outside Line · Maryland · period 9/1/25–8/31/26) from $57.60 to $57.65 · Jul 10, 2026 · via agreement page." Entries name the rate book and wage column when the agreement has them — without that, the entry is ambiguous on a split agreement. Optional "why" note; required only for ending an agreement.
Append-onlyhistory entries are never edited or deleted, by anyone.
Reconstructablebefore-and-after values on every entry mean "what did this agreement say on June 1?" is answerable by walking the history backward. With no locking, the history is the only place that answer lives — this property is what makes the no-locking decision safe.
History tab · implementation note
Shown on the agreement as a History tab: newest first, filterable by section (wages / benefits / mapping / documents / identity / lifecycle), one sentence per entry with before → after. The bones exist — the backend already snapshots before/after with a version counter on V2 edits and blocks two people from silently overwriting each other. The work: rename the concept from "amendments" to edit history, make entries field-level readable, cover the new flows (raise dates, supersede, ended, identity), and close the side doors.
